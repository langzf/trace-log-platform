# Requirement Baseline - 运维支撑体系

## 1. 背景
当前反馈处理、日志排查、缺陷定位与修复依赖人工串行处理，吞吐受限，且无法满足 7x24 连续运转需求。目标是构建内部闭环体系，自动完成“接入 -> 理解 -> 诊断 -> 修复 -> 验证 -> 复盘”主流程。

## 2. 目标与结果
- 将高频重复劳动自动化，人工聚焦最终质量把关。
- 显著缩短从问题出现到修复 PR 创建的周期。
- 建立可持续进化机制，使系统随运行持续优化。

### 北极星指标
- `MTTD`（平均发现与定位时长）
- `MTTR`（平均修复时长）
- 自动修复 PR 采纳率
- 回滚率与回归率
- 单问题综合成本（模型 + 执行 + 人审）

## 3. 干系人
- 产品/运营：关注反馈闭环效率、用户体验。
- 研发：关注根因定位准确率、修复建议质量。
- 运维/SRE：关注稳定性、发布安全、可追溯。
- 技术负责人：关注成本、可扩展性、长期演进。

## 4. 范围

### In Scope
- 多语言 SDK 接入（JS/Python/Java）和标准化事件模型。
- 缺陷分类、聚类、优先级分层。
- 日志 + Trace + 代码上下文联合诊断。
- 自动修复任务编排、执行器调度、PR 提交。
- 测试与发布门禁、回滚与复盘进化。

### Out of Scope
- 对公网用户开放的多租户 SaaS 能力。
- 外部商业计费能力。
- 对外开放 IAM 与复杂权限体系。

## 5. 约束与假设

### 假设
- 体系为内部部署，可依赖现有 Git、CI/CD、消息队列与监控平台。
- 业务系统可接入统一 SDK 或遵循事件协议。
- 研发团队接受“Agent 编码 + 人工 CR 最终把关”。

### 约束
- 成本需可控，不允许无限轮次模型调用。
- 生产发布必须具备门禁与回滚机制。
- 所有自动化任务需要完整审计链路。

## 6. 非功能需求（NFR）
- 可用性：核心链路可用性 >= 99.9%。
- 时效性：高优问题自动诊断建议在 5 分钟内输出。
- 可追溯：从事件到 PR/发布全链路可查询。
- 可扩展：执行器池支持水平扩容和多机并行。
- 可维护：策略参数（阈值、模型路由）可动态调整。

## 7. 风险
- 模型幻觉导致错误定位/修复。
- 低质量输入造成误聚类或误优先级。
- 自动修复策略过激导致无效 PR 泛滥。
- 预算控制失效导致调用成本异常。

## 8. 交付边界
本需求基线作为 PRD、架构、数据、API、详细设计的统一输入，后续任何设计变更需回写需求映射并走签署流程。
